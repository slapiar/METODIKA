# CodeIgniter 4.7.4 ako technické prostredie METODIKY

## Stav dokumentu

```text
PRACOVNÝ
```

Tento dokument zaznamenáva pracovné pochopenie filozofie frameworku CodeIgniter 4.7.4, logiky spracovania požiadaviek a pravidiel jeho použitia v projekte METODIKA.

Neurčuje nové ontologické definície METODIKY. CodeIgniter je technický nosič aplikácie; význam SUBJECT-ov, otázok, hodnotení, dôkazov, Validácie, Autority a metodických úkonov zostáva určený autoritatívnymi dokumentmi METODIKY.

---

# 1. Základná filozofia

CodeIgniter má zabezpečiť technický beh aplikácie:

```text
HTTP komunikáciu
routing
bezpečnostné filtre
konfiguráciu
spracovanie vstupov
databázový prístup
logovanie
cache
sessions
migrácie
testovanie
CLI príkazy
```

Nemá určovať význam domény METODIKY.

Základné pravidlo:

```text
CodeIgniter
= technické prostredie

METODIKA
= významová a rozhodovacia doména
```

Framework sa nesmie stať náhradou ontológie ani metodických pravidiel.

---

# 2. Tok HTTP požiadavky

Pracovný tok požiadavky:

```text
prehliadač alebo klient
→ public/index.php
→ inicializácia frameworku
→ routing
→ filtre pred controllerom
→ controller
→ aplikačná služba
→ doménový algoritmus
→ repository alebo model
→ databáza
→ výsledok aplikačnej operácie
→ view alebo API response
→ filtre po controlleri
→ HTTP odpoveď
```

Každá vrstva má odlišnú zodpovednosť. Technický tok sa nesmie zamieňať s metodickým úkonom.

---

# 3. Základná štruktúra CodeIgnitera

```text
codei/
├── app/
├── public/
├── writable/
├── tests/
├── system/
├── vendor/
├── spark
└── .env
```

Význam:

- `app/` — aplikačný kód METODIKY,
- `public/` — jediný verejne prístupný webový koreň,
- `writable/` — prevádzkové dáta, cache, sessions, logy a uploady,
- `tests/` — automatizované testy,
- `system/` — jadro frameworku; nesmie sa upravovať,
- `vendor/` — závislosti spravované Composerom; nesmú sa ručne upravovať,
- `spark` — CLI vstup CodeIgnitera,
- `.env` — lokálna a prevádzková konfigurácia; nesmie obsahovať verejne commitované tajomstvá.

Základné pravidlo zásahu:

```text
codei/system/
= neupravovať

codei/vendor/
= neupravovať ručne

codei/app/
= vlastný vývoj METODIKY
```

---

# 4. MVC nie je celá architektúra

CodeIgniter používa MVC:

```text
Model
= technická práca s dátami

View
= zobrazenie výsledku

Controller
= prijatie požiadavky a koordinácia toku
```

Controller nemá obsahovať úplnú doménovú logiku. Má zostať tenký a odovzdať prácu aplikačnej službe.

Nevhodný model:

```text
Controller
→ načíta vstupy
→ vykoná celý odvodzovací algoritmus
→ rozhodne Validáciu
→ určí Autoritu
→ zapíše históriu
→ vykreslí výsledok
```

Vhodný model:

```text
Controller
→ overí technickú použiteľnosť vstupu
→ zavolá aplikačnú službu
→ prijme výsledok
→ vráti view alebo API response
```

---

# 5. Navrhované vrstvy METODIKY

```text
HTTP vrstva
→ Controllers

Aplikačná vrstva
→ Services / Use Cases

Doménová vrstva
→ SUBJECT-y, otázky, odvodzovanie, hodnotenie, Validácia, Autorita

Dátová vrstva
→ Repositories / Models

Prezentačná vrstva
→ Views / API Responses

Technická infraštruktúra
→ Config, Filters, Database, Logging, Cache, CLI
```

Pracovný tok:

```text
HTTP požiadavka
→ Controller
→ Application Service
→ Domain Algorithm
→ Repository / Model
→ metodický výsledok
→ história
→ View alebo API Response
```

# 6. Services

Služby majú niesť opakovateľné aplikačné procesy a koordinovať doménové algoritmy.

Pracovní kandidáti:

```text
QuestionDerivationService
SubjectValidationService
EvaluationService
EvidenceService
ValidationService
AuthorityService
HistoryService
```

Služba umožní použiť rovnaký proces:

- z webového formulára,
- z API,
- z CLI,
- z automatického testu,
- z budúceho AI agenta.

Doménový algoritmus nemá byť závislý od konkrétneho controllera ani od HTML rozhrania.

---

# 7. Filtre

Filtre sa používajú na technické pravidlá pred alebo po spracovaní controllera, napríklad:

```text
autentifikácia
prístup k URL
CSRF
rate limiting
maintenance režim
technický audit požiadavky
```

Filtre nemajú rozhodovať ontologické a metodické otázky.

Treba zachovať rozlíšenie:

```text
HTTP oprávnenie
≠ Autorita metodického úkonu
```

```text
filter:
Môže používateľ pristúpiť k tejto route?

metodická služba:
Je tento ACTOR oprávnený vykonať alebo Validovať konkrétny metodický úkon?
```

---

# 8. Moduly a vnútorné členenie

CodeIgniter umožňuje modulárne členenie. METODIKA však nemá byť predčasne rozdelená na množstvo technicky izolovaných modulov, kým nie sú potvrdené ich významové hranice.

Prvé pracovné členenie môže byť:

```text
app/
├── Controllers/
├── Domain/
│   ├── Subjects/
│   ├── Questions/
│   ├── Derivation/
│   ├── Evaluations/
│   ├── Evidence/
│   ├── Validation/
│   ├── Authority/
│   └── History/
├── Services/
├── Models/
├── Views/
└── Config/
```

Až stabilná a metodicky potvrdená hranica môže byť dôvodom na vytvorenie samostatného modulu.

```text
významová hranica
→ potvrdená zodpovednosť
→ samostatný modul
```

Nie opačne.

---

# 9. Databáza, migrácie a modely

Správne poradie:

```text
potvrdená metodická identita
→ potvrdené vzťahy
→ databázová migrácia
→ repository alebo model
→ aplikačná služba
→ controller
→ view alebo API
```

Nesprávne poradie:

```text
nápad
→ ručne vytvorená tabuľka
→ dodatočné hľadanie významu stĺpcov
```

Migrácie majú byť opakovateľným historickým záznamom štruktúry databázy. Ručné zmeny databázovej schémy mimo migrácií majú byť iba výnimočné, výslovne zdôvodnené a zaznamenané.

Model je technický nástroj prístupu k dátam. Nie je totožný s ontologickou identitou.

```text
Model
≠ ontológia

riadok databázy
≠ automaticky SUBJECT

technické ID
≠ identita SUBJECT-u
```

# 10. Konfigurácia a prostredia

Treba rozlišovať:

```text
app/Config/
= štruktúrna konfigurácia aplikácie

.env
= lokálne a prevádzkové hodnoty

autoritatívne dokumenty METODIKY
= významové definície a pravidlá
```

Tajomstvá, heslá a lokálne údaje patria do `.env` alebo iného ignorovaného lokálneho konfiguračného zdroja. Nemajú byť vložené do repozitára.

Prostredia:

```text
development
≠ testing
≠ production
```

Rozdiel prostredí nesmie meniť význam metodických pravidiel. Môže meniť iba technické správanie, diagnostiku, logovanie, pripojenia a prevádzkové nastavenia.

---

# 11. CLI a Spark

CLI umožňuje spúšťať procesy bez webového rozhrania.

Príklady frameworkových príkazov:

```text
php spark routes
php spark migrate
php spark db:seed
php spark cache:clear
```

Budúce pracovné príkazy METODIKY môžu byť napríklad:

```text
php spark metodika:derive-questions
php spark metodika:validate-question
php spark metodika:check-integrity
```

CLI príkaz musí používať rovnakú aplikačnú službu a doménovú logiku ako web alebo API. Nesmie vytvárať druhú paralelnú implementáciu toho istého algoritmu.

---

# 12. Testovanie

Doménové algoritmy majú byť testovateľné bez HTTP vrstvy a bez prehliadača.

Rozlíšenie testov:

```text
unit test
→ jedna trieda alebo algoritmus

integration test
→ spolupráca služieb, repository a databázy

feature test
→ route, filter, controller a výsledná HTTP odpoveď
```

Pre algoritmy otázok je prioritou:

```text
rovnaké vstupy
→ rovnaký metodický výsledok

neúplné vstupy
→ zastavenie podľa definovanej podmienky

zmena rozmeru alebo významu
→ odmietnutie kandidáta alebo nová otázka
```

---

# 13. Prvé procesné mapovanie METODIKY

Pre odvodzovanie otázky:

```text
QuestionDerivationController
→ DeriveSpecificQuestionService
→ SubjectDefinition
→ UniversalQuestion
→ DerivationAlgorithm
→ QuestionCandidate
→ DerivationRecord
→ Repository
→ história
→ zobrazenie alebo API odpoveď
```

Controller nesmie sám rozhodovať, či kandidát metodicky spĺňa kritériá. Túto činnosť vykonáva doménový algoritmus a samostatná Validácia.

---

# 14. Kritické rozlíšenia

```text
Controller
≠ doménový algoritmus
```

```text
validácia formulára
≠ metodická Validácia
```

```text
prihlásený používateľ
≠ ACTOR s Autoritou
```

```text
HTTP oprávnenie
≠ Autorita
```

```text
frameworková udalosť
≠ automaticky metodický úkon
```

```text
databázový záznam
≠ automaticky metodická skutočnosť
```

```text
technický úspech operácie
≠ pravdivosť alebo Validácia výsledku
```

```text
chyba programu
≠ odpoveď 0 na metodickú otázku
```

---

# 15. Pracovný architektonický záver

CodeIgniter 4.7.4 bude použitý ako ľahké technické prostredie a modulárny monolit.

```text
CodeIgniter zabezpečuje:
technický tok, bezpečnosť, konfiguráciu,
databázový prístup, migrácie, testovanie a rozhrania.

METODIKA zabezpečuje:
význam predmetov, otázok, odpovedí,
dôkazov, Validácie, Autority,
histórie a rozhodovacích algoritmov.
```

Základné pravidlo ďalšieho vývoja:

> Framework má prenášať a vykonávať potvrdenú metodiku. Nesmie ju svojou technickou štruktúrou predčasne určovať ani nahrádzať.
